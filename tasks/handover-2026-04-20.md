## What the feature does (end-user facing)

When a property places an order:

1. If the property was **added to our system within the last 60 days** AND has â‰¤ 7 lifetime orders â†’ flagged **NEW** (orange row, yellow "NEW" badge).
2. If the property is on the VIP priority list for its store and is NOT new â†’ flagged **Priority #N** (blue row, priority number shown).
3. Everyone else renders plain.

Sort precedence (Existing Orders default view + BC bulk release): NEW first, then Priority ASC, then JobNo-desc / CreatedOn asc. When master flag is OFF everything reverts to the pre-feature JobNo-desc sort.

## What's admin-tunable

All on `/customer-priority` in a panel titled "Customer Priority Settings (global)":

| Control | What it does | Default |
|--------|--------------|---------|
| `Feature enabled (master)` toggle | Master kill switch. OFF â†’ UI coloring, default sort, BC sort, scheduler fields all revert to legacy. | ON |
| `Emit scheduler fields` toggle | Granular kill switch. OFF â†’ `priorityNumber` and `isNewCustomer` are blanked to null/false on the `GetAllOrderInstallationDetail` wire. Precedence for each setting: **DB row (`TR_CustomerPrioritySetting`) â†’ `appsettings.json` `FeatureFlags:*` â†’ hardcoded helper defaults.** `appsettings` remains as a deploy-time backstop if the DB row is missing.

## Scheduler integration â€” what we send and what we tested

The scheduler polls `GET /Ordering/api/Order/GetAllOrderInstallationDetail` hourly. We now send **two** additional fields per order row:

- `priorityNumber` â€” int 1..N if the property is on the VIP list for its store, else `null`.
- `isNewCustomer` â€” bool per the rule above.

We intentionally removed `orderCountForProperty` from the wire â€” it was leaking internal NANCY state.

**Verification done (local, end-to-end, not just code-reading):**
- Java harness at `/tmp/scheduler-harness/SchedulerParseHarness.java` replicates the jobrunner's `fetchWithRetry` parsing logic (csr-scheduler/jobrunner/JobImportServiceNancy.java lines 1197-1260). It logs in to NANCY, pulls all orders, and applies the exact same allowlist mapping the real jobrunner uses. **958/958 rows parse cleanly** with master ON, master OFF, and scheduler OFF states.
- The jobrunner doesn't read `priorityNumber` or `isNewCustomer` â€” its allowlist covers 13 specific fields (`JobNo`, `ShipToPropertyName`, `IsDeleted`, etc.) and ignores unknowns. Adding our two is non-breaking by the scheduler's own design.

To rerun the harness (from this machine):
```bash
cd /tmp/scheduler-harness
java -cp ".:lib/*" SchedulerParseHarness
```
Exit code 0 = pass. Exit code 2 = parse errors. Any other = infrastructure (login / network) issue.

### 2. Scheduler-side PR (separate)
NANCY is fully non-breaking. For the scheduler to actually USE the new fields (as opposed to just ignoring them), the scheduler team needs their own PR:
- DB: add `priority_number INT NULL` to `csrscheduler_nancy.job`
- `backend/app/models/schemas.py` JobInput: add `priority_number: int | None = None`
- `jobrunner/JobImportServiceNancy.java` ~line 1246: read from `result.get("priorityNumber")`, write to the new Aurora column
- Backend jobs router: add `ORDER BY priority_number asc NULLS LAST` before `.limit(150)`
- Frontend JobTable: new column with `defaultSortFieldId`, `defaultSortAsc=true`
- Gate behind a scheduler-side flag like `USE_NANCY_PRIORITY`
This is Phase 2 of the 2-phase rollout described in the DEV doc. Not urgent.

### 3. Pre-existing scheduler bug (discovered, not ours)
Scheduler importer reads `result.get("isDeleted")` (camelCase); NANCY emits `IsDeleted` (PascalCase) from this endpoint. Their "is-this-job-deleted?" detection is silently always false. Worth a ticket. Does not affect Customer Priority.

### 4. Merge conflict expectation
When merging this branch to `blue`, `OrderingRepository.cs` will conflict with `f72f90c7 Allow plank on stair`. Keep both: the stair pricing logic AND our enrichment block (it's further down in the file). Resolution is mechanical.
